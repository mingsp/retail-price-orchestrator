# 商圈比价采集调度系统产品设计简报

更新日期：2026-07-06

## 1. 产品定位

这是一个面向运营采集协同的生产控制台，不是展示型后台。

系统目标是让多台电脑、多账号、多浏览器 Profile、多门店、多类目采集任务可以被统一调度、监控、暂停、恢复、归档和复盘。

## 2. 核心用户

| 用户 | 主要任务 |
|---|---|
| 运营同事 | 查看门店采集进度、处理验证码/登录/账号异常、确认账号和浏览器窗口 |
| 技术同事 | 配置 worker、排查 CDP/Profile/网络问题、维护采集脚本 |
| 项目负责人 | 查看整体门店进度、风险次数、账号可用性、产物归档情况 |

## 3. 关键设计原则

1. **中文优先**：页面、操作、状态、提示、表单尽量使用中文；技术字段只在必要时保留英文标识。
2. **密集但清晰**：这是高频操作台，优先表格、筛选、状态色、操作按钮，不做营销式首页。
3. **账号可识别**：任何风险都必须能快速定位到设备、账号、Profile、CDP 端口。
4. **风险可闭环**：验证码、403/418、账号封禁、Profile 风险、设备/IP 风险都要能记录、通知、处理、恢复。
5. **原始数据优先**：先保存完整 raw JSONL、状态日志、风险日志，再做业务清洗和导出。
6. **默认安全**：worker 默认只心跳，不自动领取和执行采集；执行必须显式开启。

## 5. 主要操作流

### 5.1 新门店采集

1. 运营在「门店」中新增门店 URL。
2. 创建采集批次。
3. 自然打开门店并生成类目计划。
4. 批量创建类目任务。
5. worker 心跳上报账号/Profile/CDP。
6. 开启任务领取和执行。
7. worker 按类目采集，上传 JSONL 和日志。
8. 风险事件进入「风险事件」并通知人工处理。
9. 采集完成后生成业务导出。

### 5.2 风险处理

1. 系统出现 `manual_required` 或风险事件。
2. 运营通过 worker、账号、Profile、CDP 定位浏览器窗口。
3. 人工处理验证码/登录/身份核实。
4. 如果账号封禁，标记账号阻断、Profile 风险，并换号换 Profile。
5. 如果同设备多账号异常，标记设备风险并暂停分配。

## 6. 界面设计要求

- 页面语言：中文优先。
- 状态展示：中文文案 + 稳定状态色。
- 操作按钮：短中文动词，例如「暂停」「完成」「需人工」「封禁」。
- 表格列：优先展示运营可决策的信息，不把接口字段暴露为主文案。
- 技术字段：`workerId`、`accountId`、`profileId`、`CDP`、`taskId` 可以保留，用于排查。
- 表单：支持批量录入类目，后续支持“类目名 + tag + 预估商品数”。

## 7. 下一阶段产品完善

1. 类目任务创建支持 tag 和预估商品数。

5. 权限和审计：记录谁标记了账号封禁、谁恢复了任务。
